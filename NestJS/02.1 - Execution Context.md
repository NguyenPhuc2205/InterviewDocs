# Execution Context — Nền tảng của Guard, Interceptor và Filter

> NestJS cung cấp các class tiện ích giúp viết ứng dụng hoạt động được trên **nhiều loại context** khác nhau (HTTP, Microservices, WebSocket, GraphQL). Đây là nền tảng giúp Guard, Interceptor, và Filter "thông minh" hơn Middleware.
>
> Nguồn: [NestJS Docs — Execution Context](https://docs.nestjs.com/fundamentals/execution-context)

---

## 0. Nền tảng — "Nhiều context" là gì? "Arguments" là gì?

Trước khi đi vào `ArgumentsHost` và `ExecutionContext`, cần hiểu rõ **2 khái niệm nền tảng** mà cả tài liệu này dựa vào.

---

### 0.1. NestJS chạy được nhiều loại context cùng lúc

**Context** (ngữ cảnh) = **loại giao tiếp** mà ứng dụng NestJS xử lý. NestJS hỗ trợ 4 loại:

| Context | Giao thức | Dùng khi nào | Ví dụ |
|---|---|---|---|
| **HTTP** | REST API (Express/Fastify) | Client gửi request GET/POST/PUT/DELETE | Web app, mobile app gọi API |
| **WebSocket** | Socket.IO / ws | Giao tiếp 2 chiều real-time | Chat, notification, live update |
| **Microservice (RPC)** | TCP, NATS, RabbitMQ, Kafka, gRPC | Service-to-service communication | Order service → Payment service |
| **GraphQL** | HTTP + GraphQL schema | Query/Mutation linh hoạt | API cho frontend phức tạp |

**Điều đặc biệt:** NestJS có thể chạy **nhiều loại context cùng lúc trong 1 ứng dụng** — gọi là **Hybrid Application** (ứng dụng lai). Tức là **1 process Node.js** đồng thời:
- Nhận HTTP request từ client
- Nhận WebSocket message từ browser
- Nhận message từ microservice khác qua RabbitMQ

```typescript
// main.ts — Hybrid App: HTTP + Microservice + WebSocket cùng 1 app
const app = await NestFactory.create(AppModule);

// Kết nối thêm Microservice listener (RabbitMQ)
app.connectMicroservice({
  transport: Transport.RMQ,
  options: {
    urls: ['amqp://localhost:5672'],
    queue: 'orders_queue',
  },
});

await app.startAllMicroservices(); // bắt đầu lắng nghe RabbitMQ
await app.listen(3000);            // bắt đầu lắng nghe HTTP + WebSocket

// Bây giờ app đồng thời nhận:
// - HTTP request trên port 3000
// - WebSocket connections trên port 3000
// - RabbitMQ messages từ orders_queue
```

### 0.2. Mỗi context có handler riêng, nhận arguments khác nhau

**Handler** = method xử lý request/message. Mỗi loại context dùng **decorator khác nhau** để đánh dấu handler, và handler nhận **arguments khác nhau**.

#### HTTP — `@Controller` + `@Get/@Post/@Put/@Delete`

```typescript
@Controller('users')
export class UsersController {

  @Get(':id')
  findOne(@Param('id') id: string, @Req() req: Request) {
    // Handler nhận: Request, Response, NextFunction
    // NestJS extract cho bạn qua decorator @Param, @Body, @Req, @Res
    return this.usersService.findOne(id);
  }
}

// Bên trong, NestJS gọi handler với arguments: [Request, Response, NextFunction]
// Nhưng bạn ít khi truy cập trực tiếp — dùng decorator @Param, @Body...
```

#### WebSocket — `@WebSocketGateway` + `@SubscribeMessage`

```typescript
@WebSocketGateway()
export class ChatGateway {

  @SubscribeMessage('sendMessage')
  handleMessage(
    @ConnectedSocket() client: Socket,    // ← Socket client
    @MessageBody() data: { text: string } // ← data client gửi
  ) {
    // Handler nhận: [data, client]
    // KHÁC HOÀN TOÀN HTTP — không có request, không có response
    client.emit('newMessage', data);
  }
}

// Bên trong, NestJS gọi handler với arguments: [data, client]
// data = payload từ client, client = Socket connection
```

#### Microservice — `@Controller` + `@MessagePattern/@EventPattern`

```typescript
@Controller()
export class OrdersController {

  @MessagePattern('process_order')  // lắng nghe pattern 'process_order'
  processOrder(@Payload() data: OrderDto, @Ctx() context: RmqContext) {
    // Handler nhận: [data, context]
    // data = message payload
    // context = thông tin về channel, pattern, message...
    const channel = context.getChannelRef();
    channel.ack(context.getMessage()); // xác nhận đã xử lý
    return { status: 'processed' };
  }
}

// Bên trong, NestJS gọi handler với arguments: [data, context]
// data = message content, context = transport-specific info (channel, pattern)
```

#### Tóm lại — arguments khác nhau hoàn toàn

```
HTTP:           handler(request, response, next)
WebSocket:      handler(data, client)
Microservice:   handler(data, context)
GraphQL:        handler(root, args, context, info)
```

**"Arguments"** trong tài liệu này = các **tham số** mà NestJS truyền vào handler khi gọi nó. Mỗi context truyền **bộ tham số khác nhau** — đây là nguồn gốc của vấn đề.

### 0.3. Vấn đề — Guard/Filter/Interceptor cần xử lý MỌI context

Bạn viết **1 cái Guard** kiểm tra authentication. Guard này cần hoạt động được cho **tất cả** loại handler:

```typescript
@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ???): boolean {
    // Cần lấy token để xác thực user

    // Nếu đang HTTP: token nằm trong request.headers.authorization
    // Nếu đang WebSocket: token nằm trong client.handshake.auth.token
    // Nếu đang Microservice: token nằm trong data.metadata.token

    // LÀM SAO biết đang ở context nào?
    // LÀM SAO truy cập đúng object (request vs client vs data)?
  }
}
```

**Không có `ArgumentsHost`/`ExecutionContext`**, bạn phải viết **3 Guard riêng biệt**:
- `HttpAuthGuard` — chỉ cho HTTP
- `WsAuthGuard` — chỉ cho WebSocket
- `RpcAuthGuard` — chỉ cho Microservice

**Với `ArgumentsHost`/`ExecutionContext`**, bạn viết **1 Guard duy nhất** xử lý tất cả:

```typescript
@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    const type = context.getType(); // ← biết đang ở context nào

    if (type === 'http') {
      const request = context.switchToHttp().getRequest();
      const token = request.headers.authorization?.split(' ')[1];
      // xác thực token HTTP...
    }
    else if (type === 'ws') {
      const client = context.switchToWs().getClient();
      const token = client.handshake?.auth?.token;
      // xác thực token WebSocket...
    }
    else if (type === 'rpc') {
      const data = context.switchToRpc().getData();
      const token = data?.metadata?.token;
      // xác thực token Microservice...
    }
  }
}
```

Tương tự cho **Exception Filter** — lỗi xảy ra ở HTTP thì trả JSON qua `res.json()`, lỗi ở WebSocket thì gửi qua `client.emit()`, lỗi ở Microservice thì trả qua transport channel. **1 Filter duy nhất** xử lý tất cả nhờ `ArgumentsHost`.

### 0.4. Tóm lại

| Khái niệm | Nghĩa | Ví dụ |
|---|---|---|
| **Context** | Loại giao tiếp mà NestJS xử lý | HTTP, WebSocket, Microservice, GraphQL |
| **Handler** | Method xử lý request/message | `@Get()`, `@SubscribeMessage()`, `@MessagePattern()` |
| **Arguments** | Tham số NestJS truyền vào handler | HTTP: `[req, res, next]`, WS: `[data, client]` |
| **ArgumentsHost** | Class bọc quanh arguments → truy cập an toàn | `switchToHttp()`, `switchToWs()`, `getType()` |
| **ExecutionContext** | Mở rộng ArgumentsHost → biết thêm handler/controller | `getHandler()`, `getClass()` |

> 💡 **Dễ nhớ:** NestJS chạy nhiều loại context → mỗi loại có arguments khác nhau → Guard/Filter/Interceptor cần 1 cách **thống nhất** để truy cập arguments → đó là `ArgumentsHost` và `ExecutionContext`.

## 1. Tại sao cần Execution Context?

Phần 0 đã giải thích nền tảng. Bây giờ đi vào chi tiết kỹ thuật.

NestJS tạo ra 2 class trừu tượng để giải quyết vấn đề "nhiều context, arguments khác nhau":

- **`ArgumentsHost`** → bọc quanh arguments, cho phép truy cập an toàn bất kể loại context
- **`ExecutionContext`** → mở rộng từ `ArgumentsHost`, thêm thông tin về handler/controller sắp chạy

```
                ┌─────────────────────────────┐
                │       ArgumentsHost         │
                │                             │
                │  • getType()                │  ← đang ở context nào?
                │  • switchToHttp()           │  ← lấy request, response
                │  • switchToWs()             │  ← lấy client, data
                │  • switchToRpc()            │  ← lấy data, context
                │  • getArgs()                │  ← lấy tất cả arguments
                └──────────────┬──────────────┘
                               │ kế thừa (extends)
                               ▼
                ┌─────────────────────────────┐
                │     ExecutionContext         │
                │                             │
                │  • getHandler()             │  ← method nào sắp chạy?
                │  • getClass()               │  ← controller nào?
                │  + tất cả method ở trên     │
                └─────────────────────────────┘
```

## 2. ArgumentsHost — "Đang ở trong context nào?"

### Ai dùng?

**Exception Filter** — method `catch()` nhận `ArgumentsHost` làm tham số thứ 2.

```typescript
@Catch(HttpException)
export class HttpExceptionFilter implements ExceptionFilter {
  catch(exception: HttpException, host: ArgumentsHost) {
    //                             ↑ đây là ArgumentsHost
  }
}
```

### Tại sao cần?

Exception có thể xảy ra ở **bất cứ loại context nào** — HTTP, WebSocket, hay Microservice. Filter cần biết đang ở context nào để trả response đúng cách:
- **HTTP**: trả JSON qua `res.json()`
- **WebSocket**: gửi qua `client.emit()`
- **Microservice**: gửi qua `context.getChannelRef()`

Nếu không có `ArgumentsHost`, bạn phải viết filter riêng cho từng loại context — lặp code rất nhiều.

---

### 2.1. Truy cập arguments — cách đúng vs cách sai

#### ❌ `getArgs()` — tại sao không nên dùng?

```typescript
// Tưởng tượng bạn viết Filter dùng getArgs()
catch(exception: unknown, host: ArgumentsHost) {
  const [req, res] = host.getArgs();
  res.status(500).json({ message: 'Lỗi' });
}
```

Code này chạy tốt với HTTP. Nhưng khi app thêm WebSocket:

```
HTTP args:      [Request, Response, NextFunction]
WebSocket args: [WsData, Socket]
```

`args[1]` lúc là `Response`, lúc là `Socket` — gọi `.json()` trên `Socket` → **crash**.

Vấn đề gốc: `getArgs()` trả về mảng theo **thứ tự phụ thuộc context** — đổi context thì index bị sai hoàn toàn.

#### ✅ `switchToHttp()` — cách an toàn

```typescript
// An toàn với mọi context
const ctx = host.switchToHttp();
const request = ctx.getRequest<Request>();    // luôn là Request nếu đang HTTP
const response = ctx.getResponse<Response>(); // luôn là Response nếu đang HTTP
```

`switchToHttp()` **đảm bảo** trả đúng kiểu — không phụ thuộc thứ tự mảng.

---

### 2.2. Chuyển đổi giữa các context

```typescript
// ===== HTTP context =====
const httpCtx = host.switchToHttp();
httpCtx.getRequest<Request>();    // → Express Request object
httpCtx.getResponse<Response>();  // → Express Response object
httpCtx.getNext<NextFunction>();  // → next() function

// ===== WebSocket context =====
const wsCtx = host.switchToWs();
wsCtx.getData();     // → data client gửi lên
wsCtx.getClient();   // → Socket client object

// ===== Microservice (RPC) context =====
const rpcCtx = host.switchToRpc();
rpcCtx.getData();     // → message data
rpcCtx.getContext();  // → context object (channel, pattern, v.v.)
```

#### ⚠️ Gọi `switchTo` sai context thì sao?

```typescript
// Đang ở WebSocket context nhưng gọi switchToHttp()
const ctx = host.switchToHttp();
const req = ctx.getRequest(); // → undefined hoặc object rỗng
```

**Không crash ngay** — nhưng data sẽ **sai** (undefined, object rỗng). Code tiếp theo truy cập `.url`, `.headers` → lỗi runtime khó debug.

**Giải pháp:** Luôn kiểm tra `getType()` trước khi `switchTo`:

```typescript
catch(exception: unknown, host: ArgumentsHost) {
  const type = host.getType();

  if (type === 'http') {
    const ctx = host.switchToHttp();
    const response = ctx.getResponse<Response>();
    response.status(500).json({ message: 'Lỗi server' });
  }
  else if (type === 'ws') {
    const client = host.switchToWs().getClient();
    client.emit('error', { message: 'Lỗi' });
  }
  else if (type === 'rpc') {
    const ctx = host.switchToRpc();
    // xử lý lỗi cho Microservice...
  }
}
```

---

### 2.3. Kiểm tra loại context — `getType()`

```typescript
host.getType();  // → 'http' | 'rpc' | 'ws'
```

Với GraphQL — cần import kiểu riêng:

```typescript
import { GqlContextType } from '@nestjs/graphql';

if (host.getType<GqlContextType>() === 'graphql') {
  // GraphQL request
  // arguments: [root, args, context, info] — khác hoàn toàn HTTP
  const gqlHost = GqlArgumentsHost.create(host);
  const info = gqlHost.getInfo();
}
```

---

### 2.4. Tổng hợp các method của ArgumentsHost

| Method | Trả về | Mục đích |
|---|---|---|
| `getType()` | `'http'` \| `'rpc'` \| `'ws'` | Biết đang ở context nào |
| `switchToHttp()` | `HttpArgumentsHost` | Lấy `request`, `response`, `next` |
| `switchToWs()` | `WsArgumentsHost` | Lấy `data`, `client` |
| `switchToRpc()` | `RpcArgumentsHost` | Lấy `data`, `context` |
| `getArgs()` | `any[]` | Lấy tất cả arguments — ⚠️ **không nên dùng** |
| `getArgByIndex(i)` | `any` | Lấy argument theo vị trí — ⚠️ **không nên dùng** |

> ⚠️ **Không nên dùng `getArgs()` và `getArgByIndex()`** — vì chúng phụ thuộc vào thứ tự arguments của từng context. HTTP `args[0]` là request, WebSocket `args[0]` là data — đổi context là hỏng. Luôn dùng `switchTo...()`.

## 3. ExecutionContext — "Handler nào sắp chạy?"

### Ai dùng?

**Guard** (method `canActivate()`) và **Interceptor** (method `intercept()`) — nhận `ExecutionContext`.

```typescript
// Guard
@Injectable()
export class AuthGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    //                  ↑ đây là ExecutionContext
  }
}

// Interceptor
@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler) {
    //        ↑ đây là ExecutionContext
  }
}
```

---

### 3.1. ExecutionContext = ArgumentsHost + 2 method mới

```typescript
export interface ExecutionContext extends ArgumentsHost {
  getClass<T>(): Type<T>;    // Controller class đang xử lý request
  getHandler(): Function;    // Method (handler) sẽ được gọi tiếp theo
}
```

| Method | Trả về | Mục đích |
|---|---|---|
| `getHandler()` | `Function` — **tham chiếu đến method handler** | Biết method nào sắp chạy |
| `getClass()` | `Type` — **tham chiếu đến class Controller** | Biết controller nào chứa handler |
| + tất cả method của `ArgumentsHost` | | `switchToHttp()`, `getType()`, v.v. |

---

### 3.2. getHandler() và getClass() trả về gì chính xác?

```typescript
@Controller('cats')
export class CatsController {
  @Post()
  @Roles(['admin'])
  create(@Body() dto: CreateCatDto) {
    return this.catsService.create(dto);
  }
}

// Khi có request POST /cats → NestJS sẽ gọi CatsController.create()
// Trong Guard:
canActivate(context: ExecutionContext): boolean {
  const handler = context.getHandler();
  console.log(handler);       // → [Function: create] (tham chiếu đến function)
  console.log(handler.name);  // → "create" (string tên method)

  const controller = context.getClass();
  console.log(controller);      // → [class CatsController] (constructor, KHÔNG phải instance)
  console.log(controller.name); // → "CatsController"
}
```

**Điểm quan trọng:** `getClass()` trả về **class (constructor)**, **không phải instance** đang chạy. Nên:
- ✅ Đọc metadata từ decorator trên class → được
- ✅ Lấy tên class → được
- ❌ Gọi method trên đó (ví dụ `controller.someMethod()`) → **không được** (vì đó là class, không phải instance)

---

### 3.3. Tại sao 2 method này quan trọng?

Vì chúng cho phép **đọc metadata** từ decorator — điều mà **Middleware không làm được**.

```typescript
// Guard đọc @Roles('admin') gắn trên handler
const roles = this.reflector.get('roles', context.getHandler());
//                                        ↑ biết handler nào → đọc decorator trên đó

// Middleware KHÔNG có getHandler() → KHÔNG đọc được @Roles()
```

**Ứng dụng thực tế ngoài đọc Roles — dùng để logging:**

```typescript
// Interceptor ghi log controller.method() cho mọi request
@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  intercept(context: ExecutionContext, next: CallHandler) {
    const controllerName = context.getClass().name;
    const handlerName = context.getHandler().name;
    const now = Date.now();

    console.log(`→ ${controllerName}.${handlerName}()`);
    // → "CatsController.create()"

    return next.handle().pipe(
      tap(() => console.log(`← ${controllerName}.${handlerName}() — ${Date.now() - now}ms`)),
      // → "← CatsController.create() — 42ms"
    );
  }
}
```

---

### 3.4. Tại sao Middleware không có ExecutionContext?

```
Request đến
    ↓
[Tầng Express/Fastify]  ← Middleware chạy ở ĐÂY
    ↓                       NestJS CHƯA xác định route
[NestJS Routing]         ← Từ ĐÂY mới biết Controller/Handler
    ↓
[Guard / Interceptor]    ← Có ExecutionContext (getClass, getHandler)
[Pipe]                   ← Có ArgumentMetadata (kiểu param, metatype)
[Filter]                 ← Có ArgumentsHost (switchToHttp, getType)
```

Middleware chạy ở **tầng Express/Fastify** — **trước** khi NestJS routing xử lý. Lúc đó NestJS **chưa biết** request sẽ vào Controller nào, handler nào → **không có gì để cho vào `getClass()` hay `getHandler()`**.

Guard, Interceptor, Filter chạy **sau** routing → NestJS đã biết đích đến → cung cấp đầy đủ context.

**Ví dụ cụ thể — Middleware không làm được mà Guard làm được:**

```typescript
// ❌ Middleware: không biết route nào có @Roles()
function adminMiddleware(req: Request, res: Response, next: NextFunction) {
  // req.url = '/cats'
  // Không biết handler là create() hay findAll()
  // Không đọc được @Roles() trên handler
  // → Phải hardcode: if (req.url === '/cats' && req.method === 'POST') ???
}

// ✅ Guard: biết chính xác handler + đọc được decorator
canActivate(context: ExecutionContext): boolean {
  const roles = this.reflector.get(Roles, context.getHandler());
  // Biết handler là create(), có @Roles(['admin'])
  // → Kiểm tra user.role có trong ['admin'] không
}
```

## 4. Reflection và Metadata — cốt lõi của decorator

### Vấn đề

Bạn gắn `@Roles('admin')` lên handler. Guard cần **đọc giá trị `['admin']`** đó trong runtime để kiểm tra quyền. Làm sao?

NestJS cung cấp 2 cách tạo decorator gắn metadata, và `Reflector` class để đọc metadata đó.

---

### 4.1. Cách 1: `Reflector.createDecorator()` — cách khuyến nghị

Đây là cách **NestJS khuyến nghị** — type-safe, không dùng string key.

```typescript
// ===== Bước 1: Tạo decorator =====
import { Reflector } from '@nestjs/core';

// createDecorator tạo decorator type-safe — kiểu dữ liệu là string[]
export const Roles = Reflector.createDecorator<string[]>();

// ===== Bước 2: Gắn lên handler =====
@Controller('cats')
export class CatsController {

  @Post()
  @Roles(['admin'])   // ← gắn metadata ['admin'] lên handler create()
  create(@Body() dto: CreateCatDto) {
    return this.catsService.create(dto);
  }
}

// ===== Bước 3: Guard đọc metadata =====
@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {} // inject Reflector

  canActivate(context: ExecutionContext): boolean {
    // Truyền DECORATOR REFERENCE (biến Roles), không phải string
    const roles = this.reflector.get(Roles, context.getHandler());
    //                                ↑ biến        ↑ handler nào?
    // TypeScript tự biết roles có kiểu string[] | undefined

    if (!roles) return true; // không gắn @Roles → cho qua
    const { user } = context.switchToHttp().getRequest();
    return roles.includes(user.role);
  }
}
```

**Tại sao type-safe hơn?**

```typescript
// Reflector.createDecorator — TypeScript tự biết kiểu
export const Roles = Reflector.createDecorator<string[]>();

const roles = this.reflector.get(Roles, context.getHandler());
// TypeScript biết: roles có kiểu string[] | undefined
// Gõ sai tên biến → lỗi compile ngay
// Không cần tự khai báo generic type
```

---

### 4.2. Cách 2: `SetMetadata()` — cách low-level

Linh hoạt hơn nhưng dùng **string key** → dễ gõ sai, phải tự khai báo kiểu.

```typescript
import { SetMetadata } from '@nestjs/common';

// Tạo decorator thủ công bằng SetMetadata
export const ROLES_KEY = 'roles';
export const Roles = (...roles: string[]) => SetMetadata(ROLES_KEY, roles);

// Gắn lên handler
@Post()
@Roles('admin', 'moderator')
// → gắn metadata: { key: 'roles', value: ['admin', 'moderator'] }
create() { ... }

// Guard đọc bằng STRING KEY
const roles = this.reflector.get<string[]>(ROLES_KEY, context.getHandler());
//                               ↑ tự khai báo generic    ↑ dùng string key
```

**Vấn đề của string key:**

```typescript
// ROLES_KEY = 'roles'
const roles = this.reflector.get<string[]>('roles', context.getHandler()); // ✅ đúng
const roles = this.reflector.get<string[]>('role', context.getHandler());  // ❌ typo!
// → undefined — không báo lỗi compile, chỉ lỗi runtime
```

---

### 4.3. So sánh 2 cách

| | `Reflector.createDecorator()` | `SetMetadata()` |
|---|---|---|
| **Key** | Decorator reference (biến) | String key (`'roles'`) |
| **Type-safe** | ✅ TypeScript kiểm tra kiểu tự động | ❌ Phải tự khai báo generic `<string[]>` |
| **Lỗi typo** | Không thể — dùng biến | Có thể — gõ sai string không báo lỗi |
| **Khi nào dùng** | **Hầu hết trường hợp** — mặc định dùng cách này | Khi dùng với thư viện bên ngoài đã định nghĩa key sẵn |

**Khi nào vẫn phải dùng `SetMetadata`?**

Khi thư viện bên ngoài đã định nghĩa metadata key sẵn — bạn không thể tạo decorator mới cho key đó:

```typescript
// Thư viện auth dùng key 'IS_PUBLIC_KEY'
// Bạn phải đọc bằng đúng string key đó
const isPublic = this.reflector.get<boolean>('IS_PUBLIC_KEY', context.getHandler());
```

## 5. Gộp metadata từ nhiều tầng — Controller + Handler

### Tình huống

`@Roles` được gắn ở **cả controller lẫn method** — Guard lấy giá trị từ đâu?

```typescript
@Roles(['user', 'moderator'])    // ← Controller level: 2 roles
@Controller('posts')
export class PostsController {

  @Get()
  findAll() { ... }              // ← Không có @Roles riêng

  @Delete(':id')
  @Roles(['admin'])              // ← Method level: 1 role
  delete() { ... }
}
```

---

### 5.1. `getAllAndOverride` — method THẮNG hoàn toàn

Lấy giá trị ưu tiên cao nhất. Nếu handler có → dùng handler, bỏ qua controller. Nếu handler không có → lấy từ controller.

```typescript
const roles = this.reflector.getAllAndOverride(Roles, [
  context.getHandler(),  // ← kiểm tra method trước (ưu tiên cao)
  context.getClass(),    // ← nếu method không có → lấy từ controller
]);
```

**Kết quả cho từng route:**

```
findAll():  handler KHÔNG có @Roles → lấy từ controller → ['user', 'moderator']
delete():   handler CÓ @Roles(['admin']) → bỏ controller → ['admin']
```

**Dùng khi nào?** Khi muốn method **thay thế** hoàn toàn setting của controller:

```
Controller: @Roles(['user'])     → mặc định cần đăng nhập
Method:     @Roles(['admin'])    → route này cần admin, KHÔNG cần user nữa
```

---

### 5.2. `getAllAndMerge` — gộp tất cả lại

Gom giá trị từ **cả 2 tầng** vào 1 mảng.

```typescript
const roles = this.reflector.getAllAndMerge(Roles, [
  context.getHandler(),
  context.getClass(),
]);
```

**Kết quả cho từng route:**

```
findAll():  [] + ['user', 'moderator'] = ['user', 'moderator']
delete():   ['admin'] + ['user', 'moderator'] = ['admin', 'user', 'moderator']
```

**Dùng khi nào?** Khi muốn method **bổ sung thêm** vào setting của controller:

```
Controller: @Permissions(['read'])      → mặc định cần read
Method:     @Permissions(['write'])     → route này cần THÊM write
Kết quả:    ['read', 'write']           → cần cả 2 permissions
```

---

### 5.3. Edge case — cả 2 đều không có metadata

```typescript
// Handler KHÔNG có @Roles, Controller CŨNG không có @Roles
const roles = this.reflector.getAllAndOverride(Roles, [
  context.getHandler(),
  context.getClass(),
]);

console.log(roles); // → undefined (KHÔNG phải null, KHÔNG phải [])
```

**Cẩn thận khi kiểm tra:**

```typescript
// ✅ Đúng — check kiểu này
if (!roles) return true; // undefined → cho qua

// ❌ Sai — undefined không có .length
if (roles.length === 0) return true; // TypeError: Cannot read property 'length' of undefined
```

---

### 5.4. Bảng tổng hợp

| Method | Hành vi | Ví dụ kết quả | Dùng khi |
|---|---|---|---|
| `get(key, target)` | Lấy từ **1 target** duy nhất | Chỉ handler hoặc chỉ class | Không cần gộp/ghi đè |
| `getAllAndOverride(key, targets)` | Target đầu tiên có giá trị → **dùng luôn** | Handler: `['admin']` → trả `['admin']` | **Phổ biến nhất** — role/permission |
| `getAllAndMerge(key, targets)` | **Gộp** tất cả giá trị | Handler + Controller → `['admin', 'user']` | Gom tất cả permissions |

## 6. Ví dụ tổng hợp — Guard xử lý nhiều decorator

Ứng dụng thực tế: Guard vừa kiểm tra `@Public()` vừa kiểm tra `@Roles()`.

```typescript
// ===== Tạo các decorator =====
import { Reflector } from '@nestjs/core';
import { SetMetadata } from '@nestjs/common';

export const Roles = Reflector.createDecorator<string[]>();
export const IS_PUBLIC_KEY = 'isPublic';
export const Public = () => SetMetadata(IS_PUBLIC_KEY, true);

// ===== Controller =====
@Roles(['user'])
@Controller('users')
export class UsersController {

  @Public()                                // ← route công khai, không cần login
  @Get('profile/:id')
  getProfile(@Param('id') id: string) { ... }

  @Roles(['admin'])                        // ← chỉ admin mới được xóa
  @Delete(':id')
  deleteUser(@Param('id') id: string) { ... }

  @Get()
  findAll() { ... }                        // ← kế thừa ['user'] từ controller
}

// ===== Guard =====
@Injectable()
export class AuthGuard implements CanActivate {
  constructor(
    private reflector: Reflector,
    private jwtService: JwtService,
  ) {}

  canActivate(context: ExecutionContext): boolean {
    // 1. Kiểm tra @Public() — route công khai thì bỏ qua auth
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (isPublic) return true;

    // 2. Xác thực token
    const request = context.switchToHttp().getRequest();
    const token = request.headers.authorization?.split(' ')[1];
    if (!token) return false; // → 403

    try {
      request.user = this.jwtService.verifySync(token);
    } catch {
      return false; // token lỗi → 403
    }

    // 3. Kiểm tra role
    const requiredRoles = this.reflector.getAllAndOverride(Roles, [
      context.getHandler(),   // ← method level ưu tiên
      context.getClass(),     // ← controller level nếu method không có
    ]);
    if (!requiredRoles) return true; // không yêu cầu role → cho qua

    return requiredRoles.includes(request.user.role);
  }
}
```

**Flow cho từng route:**

```
GET /users/profile/123 (không có token)
  1. @Public() = true → bỏ qua auth → cho qua ✅

DELETE /users/1 (token hợp lệ, user.role = 'admin')
  1. @Public() = undefined → cần auth
  2. Token hợp lệ → user = { role: 'admin' }
  3. @Roles: handler có ['admin'] → getAllAndOverride trả ['admin']
  4. 'admin' ∈ ['admin'] → cho qua ✅

GET /users (token hợp lệ, user.role = 'user')
  1. @Public() = undefined → cần auth
  2. Token hợp lệ → user = { role: 'user' }
  3. @Roles: handler KHÔNG có → lấy từ controller ['user']
  4. 'user' ∈ ['user'] → cho qua ✅

GET /users (token hợp lệ, user.role = 'guest')
  1. @Public() = undefined → cần auth
  2. Token hợp lệ → user = { role: 'guest' }
  3. @Roles: ['user'] (từ controller)
  4. 'guest' ∉ ['user'] → chặn ❌ (403)
```

## 7. Tổng kết — Ai dùng class nào?

| Class | Dùng trong | Có gì đặc biệt |
|---|---|---|
| **ArgumentsHost** | Exception Filter (`catch()`) | `switchToHttp()`, `switchToWs()`, `switchToRpc()`, `getType()` — biết context nhưng **không biết** handler |
| **ExecutionContext** | Guard (`canActivate()`), Interceptor (`intercept()`) | Tất cả của ArgumentsHost + `getHandler()`, `getClass()` — **biết đầy đủ** handler/controller |
| **Reflector** | Guard, Interceptor (inject qua constructor) | Đọc metadata từ decorator: `get()`, `getAllAndOverride()`, `getAllAndMerge()` |

### So sánh mức độ "biết" của từng thành phần

```
Middleware          │ Chỉ biết: req, res, next()
                    │ KHÔNG biết NestJS context
                    │ KHÔNG đọc được decorator
────────────────────┤
Exception Filter    │ Biết: loại context (HTTP/WS/RPC)
(ArgumentsHost)     │ Truy cập request/response an toàn
                    │ KHÔNG biết handler nào
────────────────────┤
Guard / Interceptor │ Biết: TẤT CẢ
(ExecutionContext)  │ Context + Controller + Handler + Metadata
                    │ Đọc được decorator → quyết định cho phép/chặn/transform
────────────────────┤
Pipe                │ Biết: giá trị data + thông tin param
(ArgumentMetadata)  │ type, metatype, data name
                    │ Chỉ quan tâm đến data, không quan tâm handler
```

## 8. Tóm tắt thực chiến — cần gì thì dùng gì

```
Muốn biết đang ở HTTP hay WebSocket?     → host.getType()
Muốn lấy request/response HTTP?          → host.switchToHttp().getRequest()
Muốn biết handler nào sắp chạy?          → context.getHandler()
Muốn biết controller nào?                → context.getClass()
Muốn log tên controller.handler()?       → context.getClass().name + '.' + context.getHandler().name
Muốn đọc @Roles trên handler?            → reflector.get(Roles, context.getHandler())
Muốn method ghi đè controller?           → reflector.getAllAndOverride(Roles, [handler, class])
Muốn gộp cả method lẫn controller?       → reflector.getAllAndMerge(Roles, [handler, class])
Cả 2 đều không có metadata?              → trả về undefined (phải check !roles, không check .length)
```

## 9. Câu hỏi phỏng vấn

| Câu hỏi | Trả lời |
|---|---|
| `ArgumentsHost` là gì? | Class trừu tượng bọc quanh arguments của handler, hỗ trợ nhiều loại context (HTTP, WS, RPC). Dùng trong Exception Filter |
| `ExecutionContext` khác `ArgumentsHost` thế nào? | Kế thừa từ ArgumentsHost, thêm `getHandler()` và `getClass()` — biết handler/controller nào sắp chạy. Dùng trong Guard và Interceptor |
| Tại sao Middleware không có ExecutionContext? | Middleware chạy ở tầng Express — **trước** khi NestJS routing xác định handler. Lúc đó chưa biết đích đến |
| Tại sao không nên dùng `getArgs()`? | Thứ tự arguments phụ thuộc context. HTTP: `[req, res, next]`, WS: `[data, client]` — đổi context là index bị sai |
| `switchToHttp()` sai context thì sao? | Không crash nhưng trả undefined/object rỗng — phải check `getType()` trước |
| `getClass()` trả về instance hay class? | Trả về **class (constructor)**, không phải instance. Chỉ dùng đọc metadata, không gọi method được |
| `Reflector.createDecorator()` khác `SetMetadata()`? | `createDecorator`: type-safe, dùng biến làm key. `SetMetadata`: dùng string key, dễ typo nhưng linh hoạt hơn |
| `getAllAndOverride` khác `getAllAndMerge`? | `Override`: handler ghi đè controller (ưu tiên target đầu tiên có giá trị). `Merge`: gộp tất cả giá trị lại |
| Metadata undefined thì xử lý sao? | `getAllAndOverride` trả `undefined` (không phải `[]`). Phải check `if (!roles)`, không check `.length` |
