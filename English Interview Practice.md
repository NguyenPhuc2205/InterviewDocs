### Q6: "Explain how Node.js handles concurrency if it's single-threaded."

🇬🇧 **English:**
> "Node.js runs JavaScript on a **single main thread**, but it achieves concurrency through the **Event Loop** and **libuv**. When an I/O operation occurs — like reading a file or making a database query — Node.js doesn't block the main thread. Instead, it **delegates** the operation to either the **libuv thread pool** (for file system, crypto, DNS lookup) or the **OS's async I/O** (for networking like TCP/UDP).
>
> Once the operation completes, a **callback** is placed into the appropriate **Event Loop phase queue**. The Event Loop then picks it up and executes it on the main thread.
>
> This is why Node.js is great for **I/O-bound** applications — it can handle thousands of concurrent connections with just a single thread. However, for **CPU-intensive** tasks, you should use **Worker Threads** to avoid blocking the Event Loop."

🇻🇳 **Bản dịch:**
> "Node.js chạy JS trên **1 main thread**, nhưng đạt được concurrency nhờ **Event Loop** và **libuv**. Khi có I/O — đọc file, query DB — Node.js không block main thread, mà **ủy thác** cho **thread pool** (file, crypto, DNS) hoặc **OS async I/O** (networking). Khi xong, callback được đưa vào **phase queue** tương ứng, Event Loop lấy ra thực thi trên main thread."

---

### Q7: "What is the difference between `process.nextTick()` and `setImmediate()`?"

🇬🇧 **English:**
> "`process.nextTick()` belongs to the **microtask queue** and has the **highest priority** — it executes immediately after the current operation, before the Event Loop moves to the next phase. `setImmediate()` belongs to the **Check phase** of the Event Loop, which is a macrotask.
>
> An important caveat: recursive `process.nextTick()` can **starve the Event Loop** because microtasks are fully drained before any phase callback runs. `setImmediate()` is safer because it allows I/O callbacks to execute between iterations."

---

### Q8: "What is Dependency Injection and why is it important?"

🇬🇧 **English:**
> "**Dependency Injection** is a design pattern where a class **receives its dependencies from the outside** rather than creating them itself. Instead of a service instantiating its own database connection, the framework **injects** it through the constructor.
>
> In NestJS, the **DI container** manages this automatically. You declare a provider, and NestJS handles instantiation, lifecycle, and injection.
>
> Why it matters: it makes code **loosely coupled**, **easier to test** (you can inject mocks), and follows the **Dependency Inversion Principle** from SOLID — high-level modules don't depend on low-level modules, both depend on abstractions."

🇻🇳 **Bản dịch:**
> "**DI** là pattern mà class **nhận dependencies từ bên ngoài** thay vì tự tạo. NestJS có **DI container** tự động quản lý việc khởi tạo, lifecycle, và inject. Lý do quan trọng: code **loosely coupled**, **dễ test** (inject mocks), và tuân theo **DIP** từ SOLID."

### Q9: "Can you explain the NestJS Request Lifecycle?"

🇬🇧 **English:**
> "When a request hits a NestJS application, it goes through a specific pipeline:
>
> **First**, it passes through **Middleware** — similar to Express middleware, used for logging, parsing, etc.
>
> **Second**, **Guards** check if the request is authorized — for example, verifying a JWT token.
>
> **Third**, **Interceptors** can transform the request before it reaches the handler, or transform the response after.
>
> **Fourth**, **Pipes** validate and transform the incoming data — for example, using class-validator to check DTOs.
>
> **Fifth**, the **Route Handler** executes the actual business logic.
>
> **Sixth**, **Exception Filters** catch any errors thrown during the process and format the error response.
>
> The full order is: **Middleware → Guards → Interceptors (before) → Pipes → Handler → Interceptors (after) → Exception Filters**."

---

### Q10: "What is the Adapter Pattern? Can you give an example?"

🇬🇧 **English:**
> "The **Adapter Pattern** allows two incompatible interfaces to work together by creating a **wrapper** that translates one interface into another.
>
> In my project, I implemented this for the **Mail module**. I defined a common interface called `MailSender` with a method `sendEmail()`. Then I created two adapters: **ResendAdapter** (using the Resend API) and **SMTPAdapter** (using Nodemailer). Both implement the same `MailSender` interface, so the rest of the application doesn't care which provider is being used.
>
> Switching providers is just a matter of **changing the configuration** — no business logic needs to change. This follows the **Open/Closed Principle**: the module is open for extension (add new adapter) but closed for modification."

🇻🇳 **Bản dịch:**
> "**Adapter Pattern** cho phép 2 interface không tương thích làm việc cùng nhau bằng cách tạo **wrapper** dịch interface này sang interface kia. Trong project, tôi dùng cho **module Mail** — tạo interface `MailSender` chung, rồi 2 adapter: **ResendAdapter** và **SMTPAdapter**. Đổi provider chỉ cần đổi config, không cần sửa business logic."

## 🔵 Topic 4: Project Discussion (Thảo luận dự án)

---

### Q15: "Describe the architecture of your project."

🇬🇧 **English:**
> "My project follows a **modular monolith architecture** built with NestJS. Each feature is organized as a separate **NestJS module** — for example, Auth module, User module, Mail module, etc.
>
> Each module follows a consistent structure: it has its own **controller** for handling HTTP requests, **service** for business logic, **DTOs** validated with **Zod**, and **Prisma** for database access.
>
> For the database, I use **PostgreSQL** with **Prisma ORM**, which provides type-safe database access and easy schema migrations.
>
> Authentication uses **JWT** with an **access token / refresh token** strategy. The access token is short-lived (15 minutes) and the refresh token is stored in an **httpOnly cookie** for security.
>
> Configuration is validated at startup using **Zod schemas** — if any environment variable is missing or invalid, the app fails fast with a clear error message."

---

### Q16: "What would you do differently if you started this project again?"

🇬🇧 **English:**
> "A few things I would change:
>
> **First**, I would adopt **ESM (ES Modules)** from the beginning instead of CommonJS, since the ecosystem is moving toward ESM.
>
> **Second**, I would implement **comprehensive integration tests** earlier in the development process, not just unit tests. Integration tests with a test database catch more real-world bugs.
>
> **Third**, I would set up a **proper CI/CD pipeline** from day one — automated linting, testing, and deployment. Having this early prevents many issues from accumulating.
>
> These are lessons I learned through experience, and I'd apply them to any new project."

| English | Tiếng Việt |
|---------|------------|
| Don't say "I don't know" and stop | Đừng nói "Tôi không biết" rồi im lặng |
| Don't memorize answers word-by-word | Đừng thuộc lòng câu trả lời — nói tự nhiên |
| Don't speak too fast | Đừng nói quá nhanh |
| Don't badmouth previous employers | Đừng nói xấu công ty cũ |
| Don't lie about skills | Đừng nói dối về kỹ năng |

---

> 💡 **Mẹo cuối**: Khi chưa hiểu câu hỏi tiếng Anh, **đừng đoán mò**. Hỏi lại:
> - *"Could you repeat that, please?"*
> - *"Could you rephrase that?"*
> - *"Do you mean...?"*
>
> Người phỏng vấn sẽ đánh giá cao việc bạn hỏi lại hơn là trả lời sai.
